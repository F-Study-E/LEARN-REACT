# useRef

리액트 기본 제공 훅으로 `{current: initialValue}` 형태의 ref 객체를 반환한다.

## 1. 동작원리

첫 번째 렌더링에서 `{current: initialValue}` 객체 반환, 이 객체를 리액트가 저장해두기 때문에 이후 렌더링에서도 같은 객체를 반환한다.

```typescript
// useRef의 내부 구현 (개념적으로)
function useRef(initialValue) {
  // 💡 포인트: useState를 쓰지만 'setter'를 외부로 노출하지 않음
  const [ref, _] = useState({ current: initialValue });
  return ref;
}
```

### 💡 왜 `{current: ...}`라는 객체(상자)를 사용하는가?

자바스크립트의 **참조(Reference)** 특성을 이용하기 위해서다.

- **주소 고정:** 컴포넌트 함수가 리렌더링되어 재실행되어도, React는 힙(Heap) 메모리에 저장된 **동일한 객체 주소**를 반환한다.
- **가변성(Mutability):** 객체의 주소(껍데기)는 그대로 둔 채, 내부의 `current` 프로퍼티(내용물)만 변경하면, 리액트의 감시망을 피하면서도 값을 잃어버리지 않고 유지할 수 있다.

---

## 2. 왜 리렌더링이 일어나지 않는가?

### 핵심: `setter` 함수는 '업데이트 스케줄러'다

`useState`가 반환하는 `setter`는 단순히 값을 바꾸는 함수가 아니다. 리액트 엔진에 **"이 컴포넌트는 업데이트가 필요하니 리렌더링 스케줄에 올려줘!"**라고 신호를 보내는 트리거(Trigger) 역할을 한다.

- **`useState`:** `setter`를 호출함 → 리액트 엔진이 인지함 → 리렌더링 스케줄링 → 화면 갱신.
- **`useRef`:** `setter`를 사용하지 않음 → 리액트 엔진은 변화가 일어난 줄 모름 → 조용히 값만 바뀜.

### 리액트의 변화 감지 방식: "호출 대기형(Trigger)"

리액트는 변수를 실시간으로 감시(Observation)하지 않는다. 오직 **`setter`가 명시적으로 호출되었을 때만** "아, 바뀌었구나!"라고 판단한다. `ref.current = 1`은 리액트 입장에서는 그저 일반적인 자바스크립트 연산일 뿐이기에 리액트를 깨우지 못한다.

---

## 3. cf. state와 비교

### state는 "교체(Replacement)"다

리액트는 **불변성(Immutability)**을 기반으로 동작한다. `state`를 바꿀 때는 기존 것을 수정하는 게 아니라 **새로운 값/객체로 통째로 갈아끼워야 한다.**

- **이유:** 리액트는 성능을 위해 객체 내부를 전수조사하지 않고, **주소값 비교(Shallow Compare, `===`)**만 수행하기 때문.
- **결과:** 주소가 바뀌면(`setCount`) 리액트는 즉시 변화를 알아채고 화면을 다시 그린다.

### ref는 "유지(Retention)"다

`ref`는 주소값을 절대로 바꾸지 않는다.

```typescript
// 1. 최초 렌더링: 0x001 주소 생성
const myRef = { current: 0 };

// 2. 값 변경: 주소는 여전히 0x001
myRef.current = 10;

// 3. 리액트의 판단: "0x001 주소 그대로네? 리렌더링 안 함!"
```

## 4. 요약: 전광판 vs 비밀 수첩

| 구분            | `useState`                                        | `useRef`                                             |
| --------------- | ------------------------------------------------- | ---------------------------------------------------- |
| **비유**        | **전광판**: 내용을 바꾸면 즉시 온 동네가 알게 됨. | **비밀 수첩**: 나만 몰래 적어두고 나중에 꺼내 봄.    |
| **메커니즘**    | **Trigger**: `setter`가 리액트 스케줄러를 깨움.   | **Passive**: 엔진 모르게 값만 유지함.                |
| **데이터 처리** | **불변성**: 주소를 바꿔서 변화를 증명함.          | **가변성**: 주소는 고정하고 내부 값만 수정함.        |
| **주 용도**     | UI와 연결된 동적인 데이터.                        | DOM 접근, 타이머 ID, 이전 값 저장 등 UI와 무관한 값. |

---

**정리하며:** `ref.current`를 사용하는 것은 리액트의 리렌더링 메커니즘을 우회하여 **"렌더링 사이클에서 독립적인 메모리 공간"**을 확보하는 영리한 방법입니다.
