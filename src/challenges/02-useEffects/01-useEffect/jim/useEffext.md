# useEffect

: 리액트 컴포넌트가 렌더링된 이후 외부 시스템(API, DOM, 타이머 등)과 동기화할 수 있게 해주는 훅  
➡️ 리액트 핵심 원칙인 순수성을 유지하며, 부수 효과를 안전하게 다루기 위해 사용  
(순수하게 화면을 그리는 일을 끝내고 그 다음 바깥일을 처리하자!)

ex) 데이터 페칭. 이벤트 리스너 등록, 구독 설정 및 해제, 수동 DOM 조작

## 동작원리

`useEffect`는 컴포넌트가 UI 생성을 끝낸 뒤 실행되는 후속 작업 전용 공간임

- 브라우저 화면을 모두 그리고 실행됨 ➡️ 이펙트 내부 무거운 작업이 사용자 경험을 방해하지 않기 위해
- 각 렌더링마다 고유한 useEffect가 생성되며, 해당 렌더링 시점의 props와 state를 스냅샷처럼 기억함

<details>
  <summary>스냅샷처럼 기록한다?</summary>
  🤔 어차피 값이 바뀌면 새로 실행하는데 왜 굳이 이전 값을 기억한다는거지?  
  비동기 작업 때문!

```typescript
useEffect(() => {
  setTimeout(() => {
    console.log(count); // 여기서 찍히는 값은?
  }, 3000);
}, [count]);
```

1. count가 0일 때 이펙트 실행
2. 1초 뒤 버튼을 눌러 count를 5로 변경함(컴포넌트 리렌더링)
3. 콘솔에는 여전히 0이 찍힘

클로저 설명하려구 스냅샷이라고 했어요

- diffing: useEffect가 언제 실행되는지 이해(불필요한 실행 방지)
- 클로저: useEffect 안 데이터가 왜 그 값인지(데이터 일관성)
</details>

#### 실행 순서

컴포넌트 함수를 실행(이때 리액트는 useEffect의 의존성 배열을 통째로 메모리에 저장함)  
➡️ 리액트가 변경된 사항을 실제 DOM에 반영함  
➡️ 브라우저가 화면을 실제로 그리고 사용자는 변화된 UI를 봄  
➡️ (재렌더링의 경우) 이전 렌더링에서 생성된 클린업 함수를 실행함  
➡️ useEffect 내부 코드를 실행

### 비교하기(Diffing)

- 비교는 새로운 렌더링이 일어날 때 발생  
  ➡️ 이전 렌더링에서 [count]배열의 값이 [0]이라고 리액트가 메모리에 기록
- setter에 의해 컴포넌트가 재실행됨  
  ➡️ 이때 useEffect를 만나면 리액트는 현재 [count]값을 확인함([1])
- 리액트는 내부적으로 `Object.is()`를 사용하여 이전 값과 현재 값을 비교(Fiber에 저장해둔 값과 현재 상태 비교)  
  ➡️ Object.is(0, 1) → false (다르군)  
  ➡️ 값이 바뀌었으니 이번 페인팅이 끝나면 이펙트를 실행하자 플래그를 세워둠

### 왜 굳이 화면을 그린 뒤 실행할까?

목적이 UI와 상관없는 외부 작업이기 때문

만약 useEffect 안에서 서버 데이터를 가져오는데 2초가 걸린다면, 화면을 그리기 전 이 작업을 수행할 경우 사용자는 2초간 빈 화면을 봐야함  
➡️ 우선 화면을 먼저 보여주고(paint!), 그 뒤에 부수적 작업을 처리해 사용자 경험을 최적화

## 의존성 배열

➡️ 변화가 있을 때만 동기화할 것

리액트는 렌더링마다 모든 코드를 재실행하지만, 외부 시스템과 연결은 비용이 큼  
이를 최적화하기 위해 얕은 비교를 사용함

- [] : 컴포넌트가 마운트될 때 1회 실행
- [state]: state가 변경될 때마다 실행
- 배열 생략: 매 렌더링 시 실행(권장 ㄴㄴ)

> 이전 렌더링 의존성 값들과 현재 값을 `Object.js`로 비교  
> **[] 빈 배열의 경우**: 비교할 대상이 없음, 따라서 마운트 시 1회, 언마운트시 1회라는 특수 케이스로 동작  
> **배열이 없는 경우**: 비교 자체를 하지 않으므로 매 렌더링마다 실행

### 만약 의존성 배열에 일반 값을 넣는다면?

일반 변수 값이 바뀌는 것만으로 useEffect가 실행되지 않음

#### 🤔 리액트가 언제 의존성 배열을 검사할까?

리액트는 렌더링이 트리거되어야 검사한다

리액트는 변수 값이 변하는지 실시간으로 지켜보고 있지 않음. 리액트가 `useEffect`의 의존성 배열을 확인하는 유일한 시점은 컴포넌트가 리렌더링될 때

> - 일반 변수: 값이 바뀌어도 리액트 엔진이 아무 신호를 주지 않음 ➡️ 값이 바뀐 줄 모르고 컴포넌트를 다시 그리지도 않음 ➡️ useEffect 의존성 배열 검사를 하지 않음
> - state: set함수로 값이 바뀌면 리액트에게 값이 바뀌었다는 신호(trigger)를 보냄 ➡️ 리액트가 컴포넌트를 재실행하며 의존성 배열 안 값을 확인함 ➡️ 값이 바뀌었으면 useEffect 재실행

#### 🤔 만약 일반 변수를 의존성 배열에 넣으면?

다른 state의 변경으로 컴포넌트가 리렌더링되어 `useEffect`가 의존성 배열을 검사하더라도 일반 변수는 위험함

컴포넌트가 재실행될 때 지역 변수(`const x = 1;`)는 매번 새로 선언됨
만약 그 변수가 객체의 배열이라면, 내용은 같아도 메모리 주소가 매번 달라지므로 리액트가 이를 값 변경으로 착각하여 불필요한 useEffect를 재실행할 수 있음

## 클린업

➡️ 다음 동작을 위한 정리

이펙트 함수에서 반환하는 함수는 단순히 언마운트 시에만 실행되는 것이 아님

> 1. 다음 이펙트가 실행되기 직전 ➡️ 이전 이펙트 정리
> 2. 컴포넌트가 사라질 때 ➡️ 언마운트
>
> ➡️ 경쟁 상태(Race Condition) 방지와 메모리 누수 차단을 위해!  
>  (새로운 데이터로 동기화하기 전에 이전 연결을 끊는 리셋 역할)

### Race Condition

여러 개의 비동기 작업이 동시에 진행될 떄, 작업이 완료되는 순서가 시작된 순서와 달라 최종 결과값이 엉뚱하게 덮어씌워지는 현상

#### 🎥 발생 시나리오

1. 사용자가 '사용자 A'의 프로필을 클릭합니다. (API 호출 A 시작)
2. 사용자가 마음을 바꿔 즉시 '사용자 B'를 클릭합니다. (API 호출 B 시작)
3. 네트워크 상황: 호출 B는 50ms 만에 끝났는데, 호출 A는 서버 문제로 500ms 뒤에 끝났습니다.

결과: 화면에는 '사용자 B'가 선택되어 있는데, 데이터는 뒤늦게 도착한 '사용자 A'의 정보로 덮어씌워집니다.

#### 해결 방법 1. 불리언 플래그(Cleanup Pattern)

클린업 함수를 사용해 이전 렌더링에서 시작된 요청을 무시

```typescript
useEffect(() => {
  let ignore = false; // 1. 플래그 설정

  async function startFetching() {
    const json = await fetchUser(userId);
    if (!ignore) {
      // 2. 이펙트가 클린업되지 않았을 때만 상태 업데이트
      setUser(json);
    }
  }

  startFetching();

  return () => {
    ignore = true; // 3. 다음 렌더링이 시작되거나 언마운트되면 true로 변경
  };
}, [userId]);
```

#### 해결방법 2. AboutController(Native API)

브라우저의 AboutController를 사용해 아예 네트워크 요청 자체를 취소하는 더 깔끔한 방식

```typescript
useEffect(() => {
  const controller = new AbortController(); // 1. 컨트롤러 생성

  async function startFetching() {
    try {
      const response = await fetch(`https://api.example.com/user/${userId}`, {
        signal: controller.signal, // 2. 요청에 신호 전달
      });
      const data = await response.json();
      setUser(data);
    } catch (error) {
      if (error.name === "AbortError") {
        // 요청 취소는 에러가 아니므로 무시
      } else {
        handleError(error);
      }
    }
  }

  startFetching();

  return () => {
    controller.abort(); // 3. 이펙트 재실행 시 이전 요청 취소
  };
}, [userId]);
```

| 구분     | Boolean Flag (Ignore)                             | AbortController                                  |
| :------- | :------------------------------------------------ | :----------------------------------------------- |
| **장점** | 구현이 매우 간단함. 모든 비동기 로직에 적용 가능. | 실제 네트워크 트래픽을 줄여줌 (서버 부하 감소).  |
| **단점** | 데이터는 끝까지 받아오되 무시만 함.               | `fetch` 외의 일부 라이브러리에서 추가 설정 필요. |
| **추천** | 간단한 로직, 라이브러리 제약이 있을 때            | **최신 브라우저 환경, 성능 최적화가 중요할 때**  |
