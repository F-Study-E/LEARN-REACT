# useLayoutEffect

`useEffect`와 동일한 시그니처를 가지지만, 브라우저가 화면을 그리기 직전에 동기적으로 실행됨

## 동작원리

🤾 화면을 그리기 전에 멈춤

`useEffect`가 비동기적이라면, `useLayoutEffect`는 동기적임  
➡️ 즉 훅 내부 로직이 모두 끝날 때까지 브라우저가 화면 그리기를 멈추고 기다림

> 컴포넌트 호출, 결과물 계산  
> ➡️ 변경 사항을 실제 DOM에 반영(메모리 상 존재)  
> ➡️ useLayoutEffect 실행 - DOM은 바뀌었지만 화면은 아직 이전 상태인 찰나에 실행  
> ➡️ 브라우저가 최종 결과를 그림

#### DOM 업데이트와 브라우저 페인트

- DOM 트리 업데이트  
  자바스크립트가 DOM을 수정하는 단계, 이때 브라우저 메모리상의 DOM 트리 구조는 즉시 바뀜  
  하지만 이것만으론 우리 눈에 아무것도 보이지 않음
- 화면 그리기(paint)  
  브라우저가 변경된 DOM 트리를 바탕으로 픽셀을 계산해 모니터에 실제로 색을 입히는 과정

#### useLayoutEffect가 끼어드는 타이밍

리액트 관점에서 다시 순서를 배치해 보자

- Render phase  
  리액트가 버추얼 돔을 만들고 이전 것과 비교함
- Commit Phase  
   리액트가 계산된 변경 사항을 실제 돔에 반영  
  (이때 document.querySelector 등을 사용하면 바뀐 값을 읽을 순 있음 그러나 브라우저가 아직 페인트 하지 않았기 떄문에 사용자 눈에는 이전 화면이 보임)
- useLayoutEffect 실행  
  실제 돔은 바뀌었지만 페인트되기 전 짧은 틈에 실행  
  여기서 돔의 수치를 재서 다시 상태를 바꾸면, 브라우저는 화면을 그리려다 멈추고 수정된 최종 결과물을 가지고 단 한 번만 페인트를 수행
- Paint  
  브라우저가 화면을 그림

## 사용 이유

🤾 깜빡임을 방지하기 위해(No Flickering)

useEffect내에서 DOM 수치를 측정하고 상탤르 변경하면  
브라우저는 [원래 화면] ➡️ [그리기] ➡️ [Effect 실행] ➡️ [다시 그리기] 과정을 거침  
이때 사용자는 미세하게 튀는 현상을 보게 됨

useLayoutEffect 사용  
[원래 화면] ➡️ [측정 및 수정] ➡️ [최종 결과 그리기]가 한 번에 일어나 깜빡임이 사라짐

## 주의 - 성능 저하

useLayoutEffect는 동기적으로 실행되기 때문에 내부 로직이 무거우면 브라우저가 다음 화면을 그리지 못하고 멈춰있음

👍 DOM 측정, 애니메이션 시작 위치 계산 등 UI의 시각적 일관성이 중요한 경우에만 사용  
👎 API호출, 데이터 페칭 등 무거운 작업
