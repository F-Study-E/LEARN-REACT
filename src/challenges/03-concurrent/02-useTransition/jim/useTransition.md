# useTransition

: 상태 업데이트 우선순위를 낮춰 UI 응답성을 유지하는 훅  
→ 무거운 렌더링 작업 때문에 입력창이 버벅이거나 클릭이 씹히는 현상 해결

리액트의 모든 상태 업데이트는 기본적으로 긴급으로 처리  
`useTransition`을 사용하면 특정 업데이트를 전환(Transition) 업데이트로 분류 가능

- 전환 업데이트: 검색 결과 필터링, 대량의 차트 렌더링 등 약간 지연이 허용되는 작업

Concurrent 모드 이전엔 이 둘을 구분할 방법이 없었다~~

## 동작 원리

→ 우선순위 기반 렌더링

### 원리: Interruptible Rendering(중단 가능 렌더링)

Fiber Engine → 리액트가 작업을 아주 작은 단위(Fiber)로 쪼갬

> **Fiber**  
> : 하나의 컴포넌트나 DOM 노드에 대응하는 최소 작업 단위
> **Scheduler**  
> : 어떤 작업을 먼저 할지 결정하고, 브라우저 메인 스레드가 바쁜지 감시하는 역할

`startTransition` 내부 작업은 낮은 우선순위로 스케줄링  
리액트가 이 작업을 수행하다가도 브라우저에 입력 이벤트와 같은 우선순위가 높은 작업이 들어오면, 렌더링을 즉시 중단하고 입력부터 처리 후 돌아옴

1. `startTransition` 내부 상태 업데이트가 발생하면, 리액트는 이 작업을 낮은 우선순위로 분류, 메모리 상에서 렌더링 시작.
   - Fiber 단위로 실행(1번 fiber → 2번 fiber)
   - 리액트는 내부적으로 약 5ms 정도의 할당 시간을 가짐. 작업 단위가 끝날 때마다 할당 시간이 지났는지 확인 & 브라우저에 대기 중인 이벤트가 있는지 확인
2. 사용자가 키보드를 누르면, 브라우저 이벤트 큐에 입력 이벤트가 들어옴
   - Scheduler가 이를 감지, 현재 하던 우선순위가 낮은 렌더링 즉시 중단 → 브라우저에 제어권을 넘김(양보)
3. 브라우저는 이 찰나에 입력된 글자를 화면에 그리고 이벤트 처리
4. 급한 일을 마치면 리액트가 돌아옴
   - 입력값이 렌더링 결과에 영향을 주지 않는다면, 멈춘 지점부터
   - 입력값 때문에 조건이 바뀌었다면, 기존 낡은 작업 버리고 새로운 데이터부터 다시 그림

## 예시

```tsx
const [isPending, startTransition] = useTransition();

startTransition(() => {
  setHeavyState(newValue); // 이 업데이트는 중단 가능
});
```

`isPending`은 전환이 진행 중일 때 `true` — 로딩 인디케이터에 활용
